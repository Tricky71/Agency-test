# 📦 Инструкция по развёртыванию теста 

### Шаг 1. Перенос файлов на чистую CMS 1С-Битрикс («Стандарт» или выше)
1. Скопировать файлы кастомного кода в соответствующие директории сайта:
   * Папку `local/` в корень сайта 
   * Папку `catalog/` в корень сайта 
   * Файлы `index.php`, `urlrewrite.php` в  корень сайта.

### Шаг 2. Создание Highload-блока «Бренды»
1. Импортировать Hl-блок из /local/data_backup/brand_hl.xml
Должен создасться Hl-блок Бренды (таблица `b_hl_brands` ).

### Шаг 3. XML-импорт структуры инфоблока «Каталог»
1. Создать пустой тип инфоблоков с идентификатором `catalog_type` и названием `Магазин`.
2. Импортировать файл `/local/data_backup/cata.xml`  в созданный `catalog_type`.

### Шаг 4. Включение поддержки Символьных кодов для ЧПУ
1. В настройках созданного инфоблока  включить обязательную поддержку и транслитерацию символьного кода для элементов и разделов.

### Шаг 5. Импорт товаров
1. Обработчик находится в  /local/php_interface/cron/cml_import.php
2. Папку с файлом  импорта /local/1c_import/import.xml перенести в /upload.
3. Запустить можно на cron или вручную.
4. Логи пишутся в \local\logs\cml_import.log

### 💾 Работа кэширования и сброс данных в каталоге

1. **Механизм работы**: В проекте настроено автоматическое двухкомпонентное кэширование (`CACHE_TYPE => A`). Благодаря параметру `CACHE_FILTER => Y`, Битрикс динамически генерирует уникальный `Cache ID` для каждого состояния массива `$arrFilter`, кэшируя результаты работы асинхронного AJAX-фильтра.

2. **Автоматический сброс (Managed Cache)**: Кэш каталога тегирован маркерами `iblock_id_5` и `hlblock_table_b_hl_brands`. Инвалидация и полная очистка кэша происходят автоматически в фоновом режиме при:
   * Любых операциях создания/обновления/деактивации позиций пакетным Cron-скриптом импорта.
   * Ручном редактировании товаров или справочника брендов в административной панели.
   * Полной очистке кэша через инструмент автокэширования Битрикса.


### ⚡ Архитектура пакетного импорта (5000+ товаров)

Синхронизация реализована в итерационном (пакетном) режиме для защиты сервера от перегрузок:

* **Потоковый XML-пакет (`XMLReader`)** — Файл читается построчно. В памяти одновременно находится строго один товар. Исключены утечки RAM и ошибка `memory_limit`.
* **Шаговый сборщик мусора (`Garbage Collector`)** — Каждые 100 товаров скрипт вызывает `gc_collect_cycles()`. Потребление памяти стабильно держится в пределах 25 МБ.
* **Транзакции микро-пачками (MySQL)** — Деактивация отсутствующих в XML товаров идет пачками по 50 штук. Таблицы БД не блокируются, сайт не падает в 504 таймаут.
* **Изолированный `try-catch` для каждой записи** — Сбой или битый тег в одном товаре не прерывает импорт. Ошибка пишется в лог, скрипт продолжает обработку остальных 4999 позиций.
